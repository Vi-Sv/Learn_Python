### 1. ВХОДНОЙ СЫРОЙ КОД С СОЦ. КОММЕНТАРИЯМИ
```python
import pandas as pd

# Базовый Датафрейм поставок (Брак заложен чистыми булевыми флагами True/False без кавычек)
raw_data = [
    {"Объект": "ЖК центральный", "Поставка_тонн": 45, "Брак": False},
    {"Объект": "Объект Восточный", "Поставка_тонн": 38, "Брак": True},
    {"Объект": "Участок Северный", "Поставка_тонн": 50, "Брак": False}
]
df = pd.DataFrame(raw_data)

# создаем клише из нескольких условий
# КРИТИЧЕСКИЙ СИНТАКСИС: каждое условие ОБЯЗАНО быть изолировано круглыми скобками ().
# Оператор | (вертикальная черта) означает ИЛИ — пропускает строку, если совпало хотя бы одно условие.
complex_clishe = (df["Поставка_тонн"]<45) | (df["Брак"]==True)

print("Как выглядит сложное клише в RAM:")
print(complex_clishe)

# накладываем клише (маску) на таблицу, тем самым скрывая лишнее. 
# Исходная таблица не меняется, новая таблица с данными, сохранившимися после применения маски физически формируется в RAM
df_problematic = df[complex_clishe]

print("\nТаблица проблемных поставко (комбинированный Where)")
print(df_problematic)
```

### 3. СУХОЙ РАЗБОР ФИЗИКИ ВЗАИМОДЕЙСТВИЯ
*   **Закон Круглых Скобок `()` (Блокировка хаоса):** Побитовые операторы `&` и `|` в архитектуре Python имеют более высокий приоритет вычисления, чем математические операторы сравнения `<` или `==` [91.24]. Если убрать круглые скобки, процессор сначала попытается выполнить операцию `45 | df["Брак"]`, что мгновенно вызовет каскадный сбой и обрушит скрипт с аварией `TypeError`. Круглые скобки создают жесткие изолированные барьеры, принуждая процессор сначала вычислить логику внутри них, и только потом склеить результаты [91.24].
*   **Физика оператора `|` (Векторное ИЛИ):** Этот оператор работает как расширитель сита [91.24]. Процессор сопоставляет две булевы Series построчно. На строке `1` левое условие (`38 < 45`) дает `True`, правое (`True == True`) дает `True`. На выходе трафарет получает флаг `True` [91.24]. Строка ЖК Центральный (`45 < 45` -> `False` и `False == True` -> `False`) дает на выходе `False` и полностью отсекается [91.24].
*   **Материализация новой матрицы:** Команда `df[complex_clishe]` берет итоговый комбинированный трафарет, сканирует RAM, копирует физические строки, помеченные флагом `True`, и пакует их в абсолютно новый изолированный DataFrame `df_problematic` [91.24]. Исходный `df` остается нетронутым [91.24].

### 4. БИЗНЕС-ЦЕННОСТЬ ДЛЯ СМЕНЫ РЕМЕСЛА
Комбинированные маски фильтрации — это прямой аналог сложнейших запросов `WHERE ... AND/OR ...` из SQL-баз данных, перенесенный в плоскость оперативной памяти [0.1.2, 91.24]. 
На реальном производстве аналитику ПТО никогда не дают стерильные задачи с одним критерием отбора. Директору по логистике или руководителю проекта нужен комплексный аудит рисков. С помощью операторов `&` и `|` аналитик способен одним выстрелом накрыть терабайтные выгрузки сменно-суточных заданий и вычленить, например: *«все поставки бетона, где марка была ниже М350 И при этом время доставки превысило 2 часа»* [91.24]. Это позволяет выявлять скрытые системные сбои субподрядчиков и автоматически рассчитывать удержания по договорам за микросекунды [91.24].

### Код агрегации:

# Декомпозиция векторного слияния MERGE (LEFT JOIN)

**Дата фиксации:** 27.07.2026

**Целевая задача:** Построение автоматического коммерческого конвейера связывания таблиц в RAM компьютера. Запечатывание физики встроенного станка `pd.merge()` и выстраивание сквозного моста с оператором `LEFT JOIN` из СУБД SQL [0.1.2, 91.24].

## ЦЕХ №1: ИНИЦИАЛИЗАЦИЯ И ФОРМИРОВАНИЕ ДВУХ КОНТУРОВ ДАННЫХ

Включаем в работу сеанса библиотеку Pandas и разворачиваем в оперативной памяти две независимые матрицы: Таблицу Фактов (лог весовой) и Таблицу-Справочник (финансовые лимиты).

### Эталонный код инициализации:
```python
# Подключаем к данной сессии библиотеку pandas, придавая короткое имя для скоростного вызова
import pandas as pd

# 1. ТАБЛИЦА ФАКТОВ (Сырой лог поставок материалов с весовой по схеме DAFLIST)
delivery_log = [
    {"Объект": "ЖК Центральный", "Материал": "Бетон Б25", "Тоннаж": 45},
    {"Объект": "Объект Восточный", "Материал": "Бетон Б25", "Тоннаж": 38},
    {"Объект": "Участок Северный", "Материал": "Бетон Б30", "Тоннаж": 50}
]
df_facts = pd.DataFrame(delivery_log)

# 2. ТАБЛИЦА-СПРАВОЧНИК (Стоимость одного куба/тонны материала)
price_dict = [
    {"Материал": "Бетон Б25", "Цена_руб": 6500},
    {"Материал": "Бетон Б30", "Цена_руб": 7200}
]
df_prices = pd.DataFrame(price_dict)
```
## ЦЕХ №2: ХИРУРГИЧЕСКАЯ СТЫКОВКА И ДЕКОМПОЗИЦИЯ СТАНКА PD.MERGE()

Разбираем по тактам физику работы центрального узла слияния по правилу пошагового контроля промежуточных переменных (Закону P).
```python
### Код агрегации и слияния:

# ПО ШАГОВОМУ МЕТОДУ (ЗАКОН P): Склеиваем факты со стоимостью по общему ключу "Материал"
# Синтаксис: pd.merge(Левая_Таблица, Правая_Таблица, on="Общий_Ключ", how="Тип_Стыковки")
df_merged_report = pd.merge(df_facts, df_prices, on="Материал", how="left")

print("ОБЪЕДИНЕННАЯ КОММЕРЧЕСКАЯ ВЕДОМОСТЬ (АНАЛОГ LEFT JOIN):")
print(df_merged_report)
```

### СУХОЙ РАЗБОР ФИЗИКИ СТЫКОВКИ В СТРОКЕ pd.merge(df_facts, df_prices, on="Материал", how="left")
## Когда процессор доходит до этой строки, он запускает жестко детерминированный конвейер сопоставления адресов в RAM [91.24]:
### Определение ролей (Лево/Право): 
* Первый аргумент внутри скобок df_facts аппаратно объявляется Левой таблицей (Главным контуром) [91.24]. 
* Второй аргумент df_prices — Правой таблицей (Справочным контуром) [91.24].
### Заводка моста (on="Материал"): Параметр on= указывает процессору точное текстовое имя столбца-идентификатора, который присутствует в обеих таблицах [91.24]. 
* Этот столбец становится жестким «рельсовым путем», по которому компьютер будет сверять строки [91.24].
### Архитектура удержания (how="left"): 
*Этот ключ отдает команду: «Сохрани левую таблицу df_facts в абсолютной целости. Количество её строк (3 рейса) не должно измениться ни при каких обстоятельствах» [91.24].
### Построчный такт хэш-сопоставления:Строка 0 (ЖК Центральный): Процессор заходит в левую таблицу, видит в столбце "Материал" значение "Бетон Б25". Он мгновенно прыгает в правую таблицу df_prices и ищет в её хэш-карте точно такое же слово [91.24]. Находит строку, где "Материал" == "Бетон Б25", выковыривает оттуда значение соседнего столбца "Цена_руб" (6500) и намертво приваривает его справа к текущей строке ЖК Центрального [91.24].
### Строка 1 (Объект Восточный): Процесс повторяется. Значение "Бетон Б25" снова сопоставляется со справочником. Процессор дублирует цену 6500 и впекает её в строку Восточного [91.24].
### Строка 2 (Участок Северный): Ковш считывает значение "Бетон Б30". Компьютер прыгает в справочник, находит точное совпадение на второй строке, вытаскивает цену 7200 и монтирует её в хвост строки Северного участка [91.24].
### Выгрузка на новый адрес: Итоговая объединенная матрица записывается по новому адресу в RAM под ярлыком df_merged_report, а исходные таблицы df_facts и df_prices остаются стоять на своих местах в памяти в первозданном виде [91.24].
